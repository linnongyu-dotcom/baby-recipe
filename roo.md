# Global Rules for This Project

1. **深度阅读优先**：在对任何代码进行改动前，必须进行深度阅读并完全理解原本业务逻辑和视觉表现。
2. **严禁面目全非**：严禁为了实现新功能而大幅重构或简化原本稳定的功能逻辑。改动应遵循"最小侵入"原则。
3. **保留原始美学**：必须尊重并保留原本的视觉设计（如 CSS 变换、特定颜色、动画曲线、Tailwind 类名组合等）。
4. **图片比例保护**：所有图片在 UI 中显示时必须保持原图比例。必须使用 `object-fit: contain` 或相应的 CSS 属性，严禁产生任何维度的拉伸或形变。
5. **严禁夹带私货**：严禁在完成任务时改动任何与任务目标无关的视觉元素、文字内容或功能逻辑。保持代码纯净。
6. **遵循 Git 历史**：在修改已有逻辑或 UI 时，必须参考 `git diff` 确保不破坏原有的设计意图。
7. **绝对禁止执行破坏性回滚**：当用户要求"还原"、"回退"时，任何情况下**绝对禁止**使用 `git reset --hard` 或 `git checkout --` 等破坏性命令（无论是针对文件还是目录），否则将被视为重大违规。正确的操作应当是"向前修复"：利用 `git status` 查询变更列表，然后通过 `git show HEAD:[路径] > [路径]` 来安全地恢复历史文件内容，或者通过手动编写代码修改。
8. **禁止无实机验证的主观吹嘘**：在执行完任意改动操作（特别是 Bug 修复、样式调整、版本找回等）后，必须使用真实的验证手段（例如运行 `npm run build` 检查是否有编译错误、打开浏览器 DevTools Console 查看是否有运行时错误、使用预览工具截图等）。**绝不允许在未经工具真实验证的情况下，就主观臆断地对用户"打包票"宣称任务已完美完成**。任何未经确凿验证的结论都会被视为欺骗行为。
9. **完工前必须亲眼确认**：在回答用户"任务完成"或准备完结流程时，必须在最后一步进行**自我检查**（如检查浏览器控制台输出、查看构建结果、预览页面截图）。如果你不能看到并分析确凿的正常运行证据，绝对不能说"问题修复了"或"功能做好了"。切忌盲目乐观、瞎吹！
10. **完工前必须排查所有报错**：在宣布任务完成之前，**必须**运行 `npm run build`（或 `npm run check`）确保 TypeScript 编译无错误，同时检查浏览器控制台中是否出现了任何报错或异常。如果有报错，必须自己解决后再宣布完成。绝不允许带着满屏报错强行结束任务！
11. **维护知识库**：每当解决一个重大 Bug 或踩到极其隐蔽的"坑"时，**必须**更新项目根目录下的 `AI_KNOWLEDGE_BASE.md` 文档。记录并提炼出**总结性的技术盲区**（如依赖版本兼容性问题、Tailwind 类名冲突、React 状态管理陷阱等）。严禁记录单一的流水账或细碎琐事，确保文档是能够让 AI 快速理解项目深层逻辑的"避坑指南"。
12. **拒绝经验主义**：在排查 Bug 时，如果代码逻辑排查无果，必须跳出"逻辑编写错误"的思维定势，主动检查底层配置（构建工具配置、CSS 样式层叠、第三方依赖版本、环境变量等），确认是否存在配置冲突或版本兼容性问题。
13. **严禁用 echo 等命令敷衍写文件**：在修改文档、知识库或任何代码时，必须使用专属的文件编辑工具实打实地进行修改，以确保有完整的 Diff 记录和可见的更改历史。绝对禁止使用终端的 `echo` 等命令行手段偷懒或敷衍写入文件！否则视为严重欺骗行为。
